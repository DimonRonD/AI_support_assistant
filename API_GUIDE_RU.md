# API Guide для интеграторов

## 1) Назначение и границы API

Этот документ описывает, как внешнему приложению интегрироваться с API ассистента.

Что входит в API:
- авторизация пользователя по имени и email;
- отправка сообщений пользователя ассистенту;
- получение ответа ассистента;
- получение ручных ответов оператора поддержки для web-клиента;
- передача итоговой оценки диалога;
- передача комментария (только после оценки);
- получение состояния диалога и истории сообщений.

Что не входит в API:
- управление файлами RAG;
- администрирование кэша;
- команды очистки памяти.

## 2) Базовые параметры подключения

- Транспорт: HTTP/HTTPS
- Формат данных: `application/json`
- Базовый URL (пример): `http://localhost:8000`
- OpenAPI/Swagger: [http://localhost:8000/docs](http://localhost:8000/docs)
- Авторизация: обязательна через `POST /auth`

## 3) User Journey (сценарий интеграции)

### Этап A: старт или продолжение диалога
1. Ваше приложение создаёт или переиспользует стабильный `session_id` для пользователя.
2. Приложение вызывает `POST /auth` с `name`, `email`, `session_id`.
3. API возвращает результат авторизации и историю прошлой сессии (если найдена).
4. Приложение предлагает пользователю продолжить историю.
5. Если пользователь отказывается, приложение отправляет в `POST /text` сообщение `очистить историю`.
6. После этого продолжается обычный обмен через `POST /text`.

### Этап B: обработка перехвата поддержки/закрытого диалога
При вызове `POST /text`:
- если диалог передан поддержке, API вернёт сообщение об ожидании сотрудника;
- если диалог завершён, API вернёт просьбу поставить оценку.

### Этап C: завершение и сбор обратной связи
1. Приложение отправляет оценку `1..5` в `POST /rate`.
2. При необходимости отправляет комментарий в `POST /comment`.
3. Для отображения итогов и истории запрашивает `GET /dialog/{session_id}`.

### Этап D: ручной ответ сотрудника поддержки для web-пользователя
1. Сотрудник отправляет `/reply <session_id> <текст>` в support-чате Telegram.
2. Сообщение сохраняется в диалоге как `actor = support`.
3. Web-клиент опрашивает `GET /support/inbox/{session_id}?after_id=N` и показывает новые ответы оператора пользователю.

## 4) Эндпоинты

### `GET /help`
Возвращает текстовый список API-команд.

Пример ответа:
```json
{
  "message": "API команды: ..."
}
```

### `POST /auth`
Авторизация пользователя перед работой с диалогом.

Запрос:
```json
{
  "session_id": "crm-user-00042",
  "name": "Ivan Petrov",
  "email": "ivan.petrov@example.com"
}
```

Ответ:
```json
{
  "message": "Авторизация выполнена.",
  "has_previous_history": true,
  "history": [
    {
      "actor": "user",
      "content": "Здравствуйте",
      "created_at": "2026-05-08 10:01:05"
    },
    {
      "actor": "ai",
      "content": "Здравствуйте! Чем могу помочь?",
      "created_at": "2026-05-08 10:01:06"
    }
  ]
}
```

Примечания:
- попытки авторизации сохраняются в логах (`session_id`, `name`, `email`, timestamp);
- после успешной авторизации сессии разрешены `/text`, `/rate`, `/comment`, `/dialog/{session_id}`.

### `POST /text`
Отправляет одно сообщение пользователя в ассистент.

Запрос:
```json
{
  "session_id": "crm-user-00042",
  "query": "Как восстановить пароль?"
}
```

Ответ:
```json
{
  "answer": "Текст ответа ассистента",
  "from_cache": false
}
```

Примечания:
- в контекст включаются последние 5 сообщений памяти;
- используется RAG-контекст;
- ответ может быть возвращён из кэша;
- сообщения диалога логируются с актором (`user`/`ai`/`support`).
- если `query` ровно `очистить историю`, история текущей сессии очищается.

### `POST /rate`
Сохраняет итоговую оценку диалога.

Запрос:
```json
{
  "session_id": "crm-user-00042",
  "rating": 5
}
```

Ответ:
```json
{
  "message": "Оценка сохранена. При желании добавьте комментарий: /comment <текст>."
}
```

Валидация:
- `rating` должен быть целым числом в диапазоне `1..5`.

### `POST /comment`
Сохраняет комментарий к оценке.

Запрос:
```json
{
  "session_id": "crm-user-00042",
  "comment": "Вопрос решился быстро."
}
```

Ответ:
```json
{
  "message": "Комментарий сохранен. Спасибо за обратную связь."
}
```

Валидация:
- комментарий принимается только после установки оценки.

### `GET /dialog/{session_id}`
Возвращает метаданные диалога и ленту сообщений.

Пример ответа:
```json
{
  "dialog": {
    "session_id": "crm-user-00042",
    "status": "closed",
    "controlled_by": "support",
    "rating": 5,
    "comment": "Спасибо",
    "created_at": "2026-05-08 10:01:00",
    "updated_at": "2026-05-08 10:20:00",
    "closed_at": "2026-05-08 10:19:00"
  },
  "messages": [
    {
      "actor": "user",
      "content": "Здравствуйте",
      "created_at": "2026-05-08 10:01:05"
    },
    {
      "actor": "ai",
      "content": "Здравствуйте! Чем могу помочь?",
      "created_at": "2026-05-08 10:01:06"
    },
    {
      "actor": "support",
      "content": "Подключился оператор, уточните номер заказа.",
      "created_at": "2026-05-08 10:05:12"
    }
  ]
}
```

### `GET /support/inbox/{session_id}?after_id=N`
Возвращает новые сообщения оператора поддержки для web/API-пользователя.

Использование:
- оператор отвечает командой `/reply` в Telegram;
- web-клиент периодически забирает новые сообщения через этот endpoint.

Пример ответа:
```json
{
  "session_id": "web-6f58927ca7464bc5",
  "after_id": 0,
  "last_id": 142,
  "dialog_closed": true,
  "messages": [
    {
      "id": 141,
      "actor": "support",
      "content": "Добрый день! Подключился оператор поддержки.",
      "created_at": "2026-05-08 16:12:02"
    },
    {
      "id": 142,
      "actor": "support",
      "content": "Уточните, пожалуйста, номер заказа.",
      "created_at": "2026-05-08 16:12:20"
    }
  ]
}
```

## 5) Error Handling (обработка ошибок)

### Используемые HTTP-коды
- `200 OK`: успешный запрос
- `400 Bad Request`: нарушение бизнес-правил/валидации
- `422 Unprocessable Entity`: некорректный JSON или отсутствуют обязательные поля
- `500 Internal Server Error`: непредвиденная ошибка на стороне сервера

### Частые ошибки API

1) `POST /rate` с недопустимой оценкой
```json
{
  "detail": "rating должен быть от 1 до 5."
}
```

2) `POST /comment` до установки оценки
```json
{
  "detail": "Сначала установите оценку командой /rate <1-5>."
}
```

3) Неверный тип данных в payload (например, строка вместо числа)
- FastAPI вернёт `422` с деталями по полям.

4) Вызов методов без авторизации
```json
{
  "detail": "Сначала выполните авторизацию через POST /auth (name, email, session_id)."
}
```

### Рекомендации для интегратора
- Повторяйте запросы только для временных ошибок (`5xx`, сетевые таймауты).
- Не повторяйте автоматически `4xx`; сначала исправьте данные запроса.
- Логируйте `session_id`, endpoint и код ответа для наблюдаемости.

## 6) Best Practices для `session_id`

### Обязательные свойства
- стабильный в рамках одного пользовательского диалога;
- уникальный в рамках вашей интеграции;
- детерминированно воспроизводимый из вашего пользовательского идентификатора.

### Рекомендуемый формат
- префикс системы + уникальный ключ пользователя, например:
  - `crm-user-00042`
  - `tg-67382910`
  - `web-9f3a2c18`

### Практические правила
- используйте один `session_id` на один пользовательский поток диалога;
- не переиспользуйте `session_id` между разными пользователями;
- храните явное соответствие (`external_user_id -> session_id`);
- не помещайте в `session_id` персональные данные в открытом виде.

## 7) Примеры интеграции (`curl`)

### Авторизовать пользователя
```bash
curl -X POST "http://localhost:8000/auth" ^
  -H "Content-Type: application/json" ^
  -d "{\"session_id\":\"crm-user-00042\",\"name\":\"Ivan Petrov\",\"email\":\"ivan.petrov@example.com\"}"
```

### Отправить сообщение пользователя
```bash
curl -X POST "http://localhost:8000/text" ^
  -H "Content-Type: application/json" ^
  -d "{\"session_id\":\"crm-user-00042\",\"query\":\"Какие сроки возврата?\"}"
```

### Очистить историю общения
```bash
curl -X POST "http://localhost:8000/text" ^
  -H "Content-Type: application/json" ^
  -d "{\"session_id\":\"crm-user-00042\",\"query\":\"очистить историю\"}"
```

### Установить оценку
```bash
curl -X POST "http://localhost:8000/rate" ^
  -H "Content-Type: application/json" ^
  -d "{\"session_id\":\"crm-user-00042\",\"rating\":4}"
```

### Отправить комментарий
```bash
curl -X POST "http://localhost:8000/comment" ^
  -H "Content-Type: application/json" ^
  -d "{\"session_id\":\"crm-user-00042\",\"comment\":\"Все понятно\"}"
```

### Получить снимок диалога
```bash
curl -X GET "http://localhost:8000/dialog/crm-user-00042"
```

### Получить новые ответы оператора поддержки
```bash
curl -X GET "http://localhost:8000/support/inbox/web-6f58927ca7464bc5?after_id=0"
```
