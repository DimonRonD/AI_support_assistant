Проект Nutribot (n8n) — подробное описание для RAG-базы
1) Общее назначение проекта
Nutribot — это Telegram-бот, реализованный как workflow в n8n, который автоматизирует базовые консультационные сценарии в нише питания и тренировок.
Проект предназначен для снижения ручной нагрузки на специалистов (тренеров, нутрициологов, менеджеров), ускорения ответа пользователям и масштабирования сервиса без пропорционального роста команды.

2) Техническая составляющая
Архитектурный формат
Проект реализован как экспорт одного workflow-файла n8n (*.json).
Логика сосредоточена в:
стандартных n8n-нодах (trigger/switch/send/read/write),
встроенных Code-нодах (JavaScript) для бизнес-правил.
Основные подсистемы
Вход событий: Telegram Trigger (входящие сообщения пользователя).
Маршрутизация сценариев: анализ команд и переход в нужную ветку (/survey, /nutrition, /exercise, /giga).
Хранение состояния пользователя: Google Sheets (например, профиль пользователя и прогресс анкеты).
Генерация рекомендаций: AI-провайдеры (OpenAI и GigaChat, в разных ветках).
Обработка ошибок: ссылка на errorWorkflow (внешний workflow для error handling, если настроен в n8n).
Данные и состояния
Хранится chat_id Telegram и анкета пользователя (возраст, пол, вес, рост и пр.).
Реализована логика “шага анкеты” (FSM-подобный процесс: вопрос -> валидация -> следующий вопрос).
Для тренировочной ветки есть преобразование данных из табличного формата в сообщение пользователю.
3) Используемые технологии
Основной стек
n8n — оркестрация интеграций и бизнес-логики.
Telegram Bot API — канал взаимодействия с пользователем.
Google Sheets API (через n8n credentials) — хранение и чтение структурированных данных.
OpenAI API — генерация текстовых рекомендаций (питание и/или ответы).
GigaChat API — альтернативная/дополнительная LLM-ветка.
Вспомогательные элементы
JavaScript внутри Code-нод n8n.
HTTPS webhook для Telegram.
DNS-домен и SSL-сертификат для публичной доступности webhook URL.
4) Процесс работы (runtime-flow)
Пользовательский путь
Пользователь пишет боту в Telegram.
Telegram Trigger принимает событие.
Workflow определяет тип запроса:
старт анкеты/продолжение анкеты,
запрос питания,
запрос тренировок,
запрос через GigaChat.
При необходимости обновляется состояние в Google Sheets.
Для генеративных сценариев вызывается LLM (OpenAI/GigaChat).
Результат форматируется и отправляется пользователю в Telegram.
Пример веток
/survey: поэтапный опрос + валидация ввода + запись результата.
/nutrition: генерация рекомендаций по питанию на основании данных.
/exercise: чтение тренировочного плана из таблицы + форматирование ответа.
/giga: альтернативная генерация через GigaChat.
5) Процесс развёртывания
Предусловия
Развернутый экземпляр n8n (cloud/self-hosted).
Публичный домен с корректным DNS и SSL.
Доступные credentials:
Telegram bot token,
Google Sheets OAuth/Service account,
OpenAI API key,
GigaChat credentials.
Корректно заданный WEBHOOK_URL для n8n (базовый URL инстанса, HTTPS).
Шаги
Импортировать workflow JSON в n8n.
Привязать/создать credentials для всех интеграций.
Проверить node references на листы/таблицы и внешние workflow.
Активировать workflow.
Проверить регистрацию webhook в Telegram (getWebhookInfo).
Выполнить smoke-тесты по основным командам.
Критично для Telegram
Домен webhook должен резолвиться извне.
Нужен HTTPS.
При изменении домена/пути webhook лучше переустанавливать webhook (delete/set) и реактивировать trigger.
6) Инструкция по работе (операционная)
Для администратора/оператора
Старт: проверить, что workflow активен.
Мониторинг: смотреть executions в n8n (ошибки/таймауты/API limits).
Поддержка данных: контролировать структуру Google Sheets (названия листов/колонки/формат).
Обновления: изменения логики выполнять в workflow, экспортировать новую версию JSON.
Инциденты Telegram: проверять getWebhookInfo, DNS, SSL, доступность домена.
Для разработчика
Рекомендуется хранить workflow в Git как “исходник интеграции”.
Фиксировать changelog по версиям workflow.
Отделять production-экспорт и sanitized-экспорт (без чувствительных метаданных, где возможно).
Иметь тестового Telegram-бота и тестовую таблицу для безопасной проверки изменений.
Для службы сопровождения
Использовать тест-кейсы:
новый пользователь проходит анкету,
запрос питания после анкеты,
запрос тренировки,
fallback на неизвестную команду.
Проверять корректность валидации пользовательского ввода.
7) Особенности проекта
Low-code + custom-code hybrid: основа в n8n, тонкая логика в JS-нодах.
Один workflow как единая точка логики: упрощает старт, но требует дисциплины версионирования.
Многоканальная AI-логика: поддержка нескольких LLM-провайдеров.
Состояние в Google Sheets: быстрый запуск без отдельной БД, удобно для MVP/SMB.
Зависимость от внешних API: Telegram, Google, AI-провайдеры, значит важны ретраи/лимиты/наблюдаемость.
8) Экономические выгоды
Прямая операционная экономия
Сокращение ручной первичной консультации (опрос/базовые рекомендации).
Снижение нагрузки на специалистов и операторов.
Меньшая стоимость обработки одного запроса.
Масштабирование
24/7 обслуживание без расширения штата “по линейке обращений”.
Возможность обслуживать больше клиентов тем же составом команды.
Коммерческий эффект
Быстрее ответ пользователю -> выше вовлеченность.
Персонализация -> выше удержание и повторные взаимодействия.
Быстрее запуск новых продуктовых гипотез (новые ветки workflow) -> короче time-to-market.
IT-экономика
n8n и интеграционный подход дают более низкий порог изменений по сравнению с кастомной backend-разработкой “с нуля”.
Изменения можно вносить быстро и итеративно.
9) Ограничения и риски (важно для RAG-ответов)
Возможны ошибки webhook из-за DNS/SSL/доступности домена.
При экспорте workflow могут “утекать” технические метаданные (id, имена credential references).
Отсутствие строгого CI/CD для n8n-экспорта увеличивает риск конфигурационных ошибок.
Google Sheets как хранилище ограничен по надежности и масштабируемости для enterprise-нагрузок.
Зависимость от стабильности и лимитов сторонних API.
