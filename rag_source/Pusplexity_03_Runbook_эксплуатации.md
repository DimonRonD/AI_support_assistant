# Pusplexity — Runbook эксплуатации

## Назначение документа

Этот runbook описывает стандартные действия для запуска, проверки, сопровождения и восстановления сервиса `Pusplexity`.

## 1. Предусловия

- Доступ к репозиторию проекта.
- Настроенный `.env`.
- Для Telegram-режима: валидный `TELEGRAM_BOT_TOKEN`.
- Для web-режима: заполненный `FLASK_SECRET_KEY` и `users.txt`.
- Для всех AI-сценариев: валидный `OPENAI_API_KEY`.

## 2. Быстрый старт

### Локальный запуск

1. Установить зависимости: `pip install -r requirements.txt`.
2. Скопировать переменные: `cp .env.example .env`.
3. Запуск:
   - Web: `python web_app.py`
   - Telegram: `python bot.py telegram`
   - CLI: `python bot.py cli`

### Docker Compose запуск

1. Подготовить `.env`.
2. Добавить пользователей в `users.txt`.
3. Запустить: `docker compose up -d`.
4. Проверить: `docker compose ps`.

## 3. Проверки после запуска (smoke tests)

### Web

- Открыть `/health` и убедиться, что ответ `200 OK`.
- Войти в `/login`.
- Отправить короткий текстовый запрос.
- Проверить отправку изображения и получение ответа.

### Telegram

- Выполнить `/start`.
- Переключиться `/text` и задать тестовый вопрос.
- Переключиться `/create`, отправить короткий prompt.

### RAG

1. Включить загрузку (`/rag_add` или кнопка `RAG Add`).
2. Загрузить документ.
3. Выполнить индексацию (`/rag_index`).
4. Включить `rag_text` и задать вопрос по документу.

## 4. Операционные задачи

### 4.1 Обновление на сервере

1. Перейти в каталог проекта.
2. Выполнить `git pull`.
3. Выполнить `docker compose up -d --build`.
4. Проверить контейнеры и логи:
   - `docker compose ps`
   - `docker compose logs --tail=100 pusplexity-web`

### 4.2 Ротация/очистка данных

- Пользовательские данные: автоочистка по `USER_DATA_RETENTION_DAYS`.
- Логи действий: автоочистка по `LOG_RETENTION_DAYS`.
- RAG-источники удаляются штатно через `rag_delete`.

### 4.3 Проверка хранилища RAG

- Список источников: `rag_list`.
- Просмотр чанков консолью:
  - `python rag_view_chunks.py --list`
  - `python rag_view_chunks.py --source <имя_источника> --limit 10`

## 5. Инциденты и действия

### Инцидент: 401/разлогин в web

Симптомы:

- API возвращает `Требуется авторизация`.
- UI сообщает об истекшей сессии.

Действия:

1. Проверить cookie-сессию браузера.
2. Повторить вход.
3. Проверить корректность `FLASK_SECRET_KEY`.

### Инцидент: 403 CSRF

Симптомы:

- Ответ API: `Неверный CSRF-токен`.

Действия:

1. Обновить страницу.
2. Повторить запрос.
3. Проверить, что фронтенд отправляет `X-CSRF-Token`.

### Инцидент: 413 file too large

Симптомы:

- Ошибка: превышен лимит файла.

Действия:

1. Проверить размер файла (лимит 50 МБ).
2. Сжать/разделить файл.

### Инцидент: RAG не находит документы

Симптомы:

- `Хранилище RAG пусто` или отсутствуют релевантные ответы.

Действия:

1. Убедиться, что документ реально загружен в `data/`.
2. Повторить `rag_index`.
3. Проверить источники через `rag_list`.
4. Уточнить формулировку вопроса.

### Инцидент: Ошибки OpenAI API

Симптомы:

- Ошибки в text/image режимах.
- moderation block.

Действия:

1. Проверить валидность `OPENAI_API_KEY`.
2. Проверить доступность API.
3. Для moderation: переформулировать prompt.
4. Проверить логи `action_logs.db` и container logs.

## 6. Мониторинг и отчеты

Доступные встроенные метрики:

- число запросов в сутки;
- суммарные токены;
- распределение по компонентам (`text`, `rag`, `image_edit`, `create` и т.д.);
- уникальные пользователи web/telegram.

Инструменты:

- команда `logs` в Telegram/Web;
- CSV-выгрузка за выбранную дату.

## 7. Резервное копирование (рекомендации)

Критичные данные:

- `chroma_db/`
- `data/`
- `user_data.db`
- `action_logs.db`
- `users.txt`
- `.env`

Рекомендуется ежедневный backup томов и конфигурации.

- [ ] `.env` заполнен и валиден.

- [ ] `/health` отвечает `200`.
- [ ] RAG-индексация проходит без ошибок.
- [ ] Логи и статистика доступны.
- [ ] Бэкапы выполняются по расписанию.
