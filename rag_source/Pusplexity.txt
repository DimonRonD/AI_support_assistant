Проект Pusplexity

1) Техническая составляющая
Pusplexity — это мультимодальная система с двумя интерфейсами (Telegram-бот + Web UI), которая объединяет:

генерацию и редактирование изображений через OpenAI Image API;
текстовый чат (в том числе анализ изображения и диалог с памятью);
RAG-подсистему на ChromaDB для ответов по загруженным документам;
аудит действий и usage-метрик в SQLite.
Архитектурно проект разделён на модули:

bot.py — Telegram-бот, режимы, команды, загрузка файлов, оркестрация сценариев.
web_app.py — Flask-приложение с API (/api/command, /api/send, /api/upload, /api/rag_delete), сессиями, безопасностью и UI.
processor.py — ядро интеграции с OpenAI (image edit, image generate, text-only, text+image, text+RAG context).
rag_store.py — загрузка документов, чанкинг, эмбеддинги, индексация и поиск в ChromaDB.
user_db.py — персистентная память пользователей (SQLite user_data.db).
action_logs.py — событийный аудит (SQLite action_logs.db) + статистика/CSV-выгрузка.
auth.py — простая авторизация web по users.txt.
templates/*.html — клиентский интерфейс чата, логина, рендер Markdown.
Ключевые технические ограничения/правила:

До 10 изображений на обработку в image-режимах.
В текстовом режиме — максимум 1 изображение на запрос.
Максимальный размер входящего файла в Web API: 50 МБ.
Память диалога: 20 последних сообщений (для /text и /rag_text).
Ограничение длины промптов: 4000 символов (1000 для DALL-E совместимого create-режима).
2) Используемые технологии
Backend и инфраструктура:

Python 3.10+ (в Docker — python:3.13-slim)
Flask + Werkzeug
python-telegram-bot (polling)
gunicorn[gevent] для web-сервиса в production
python-dotenv (конфигурация через .env)
Docker / docker-compose
Traefik (HTTPS-роутинг через labels)
AI/ML и документы:

openai SDK
Модели изображений: gpt-image-1, gpt-image-1.5, gpt-image-2-2026-04-21, gpt-image-1-mini (+ алиасы совместимости)
Текстовая модель по умолчанию: gpt-5.5
Эмбеддинги: text-embedding-3-small
Векторное хранилище: chromadb (persistent local store)
Парсинг документов: pypdf, openpyxl, python-docx
Pillow для нормализации/валидации изображений
Хранение данных:

SQLite (user_data.db, action_logs.db)
Volumes для данных и индексов (/app/data, /app/chroma_db, /app/db, /data)
3) Процесс работы (runtime flow)
3.1 Telegram/Web: общий пользовательский сценарий
Пользователь выбирает режим (text, image*, create, rag_*).
Загружает данные (изображения или документы) и отправляет запрос.
Система валидирует вход, при необходимости сохраняет во временное/постоянное хранилище.
В зависимости от режима вызывается соответствующий метод ImageProcessor или RAGStore.
Ответ возвращается пользователю (текст, изображение, источники RAG, usage).
Действия логируются в action_logs.db (этапы user_request -> ai_request -> ai_response).
3.2 Текстовый режим
Поддерживает:
чистый чат;
вопрос + 1 изображение (визуальный анализ);
вопрос + ранее загруженный документный контекст.
Хранит короткую историю диалога (20 сообщений) для контекстности ответов.
Логирует токены и компонент text.
3.3 Режим RAG
Документы кладутся в data/.
При индексации:
извлекается текст;
делится на чанки (по умолчанию 500 символов, overlap 50);
считаются эмбеддинги;
записываются в коллекцию ChromaDB pusplexity_rag.
При query:
формируется embedding вопроса;
выбираются top-N релевантных чанков;
контекст собирается и отправляется в LLM;
пользователю дополнительно отдаётся список источников со score.
3.4 Режимы изображений
Edit-сценарий: 1–10 изображений + текстовый промпт -> images.edit.
Create-сценарий: текст без изображений -> images.generate.
Система нормализует входные файлы (MIME/формат), обрабатывает ошибки moderation и возвращает понятные сообщения.
4) Процесс развёртывания
Локально (быстрый старт)
Установить зависимости: pip install -r requirements.txt
Создать .env из .env.example
Запуск:
Web: python web_app.py
Telegram: python bot.py telegram
CLI: python bot.py cli
Production через Docker Compose (рекомендуемый)
docker compose up -d (или отдельный сервис, например pusplexity-web)
Оба сервиса (pusplexity и pusplexity-web) используют один Docker-образ, но разные команды запуска.
Данные сохраняются в Docker volumes:
документы RAG,
база Chroma,
SQLite (логи/память),
persistence Telegram-бота.
HTTPS за Traefik
Для web-сервиса заданы Traefik labels (router/service/tls/certresolver).
Достаточно указать корректный TRAEFIK_CERT_RESOLVER в .env.
Обновление на сервере
git pull
docker compose up -d --build
проверка статуса/логов контейнеров
5) Инструкция по работе (для конечного пользователя)
Telegram
Основные команды:
/text — чат/анализ 1 фото/контекст документа
/image1, /image15, /image2, /dalle — редактирование изображений
/create, /dalle_gen — генерация по тексту
/rag_add -> загрузить документы
/rag_index -> индексировать
/rag_text -> задавать вопросы по базе
/rag_list, /rag_delete, /rag_clear — администрирование RAG
/logs — статистика + CSV
/clear — очистка истории text-режима
Web
Войти по email/паролю (из users.txt).
Выбрать режим в панели.
Загрузить файл(ы) или отправить текст.
Для RAG:
RAG Add -> загрузка документов,
RAG Index -> индексация,
RAG -> вопросы по базе.
При необходимости скачать CSV-логи из режима Logs.
6) Особенности проекта
Единая бизнес-логика для Telegram и Web (минимум расхождений в поведении).
Гибридный мультимодальный UX: text, image-edit, image-generate, RAG.
Прозрачный аудит действий с раздельной статистикой по каналам (web/telegram).
Защита web-слоя:
CSRF-токены,
rate limiting API и логина,
security headers + CSP,
валидация типов загружаемых файлов,
ограничение размера запросов.
Персистентная пользовательская память в SQLite, но с контролем хранения (USER_DATA_RETENTION_DAYS, LOG_RETENTION_DAYS).
Совместимость моделей через алиасы и мягкие миграции legacy-значений.
7) Экономические выгоды
Снижение операционных затрат на поддержку: часть запросов пользователей закрывается автоматически через RAG и чат.
Ускорение обработки визуальных задач: редактирование/генерация изображений без ручного труда дизайнеров на типовых задачах.
Переиспользование знаний компании: документы превращаются в поисковую базу, снижается время поиска информации.
Контроль расходов на AI: токены и компоненты логируются, можно анализировать cost drivers и оптимизировать сценарии.
Масштабируемость с низким порогом: Docker/Traefik упрощают перенос между средами и эксплуатацию.
Снижение time-to-answer: единая точка входа (Telegram + Web) повышает доступность сервиса для разных ролей.
