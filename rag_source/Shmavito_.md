# Shmavito: обзор проекта для RAG

## Краткое описание

`Shmavito` - дипломный веб-проект формата C2C/C2B-маркетплейса аренды вещей (по аналогии с Avito/eBay), реализованный как монолит на Django.

Проект уже включает не только авторизацию, но и основные бизнес-сценарии:
- каталог и поиск объявлений;
- создание/редактирование товаров и предложений;
- оформление и отмена заказов с управлением интервалами дат;
- модерация контента;
- рейтинг и комментарии.

## Техническая составляющая

- Тип решения: монолитное серверное веб-приложение (`Django` + шаблоны).
- Точка входа: `manage.py` (`DJANGO_SETTINGS_MODULE=core.settings`).
- Основное приложение: `shmavito`.
- Доменная модель: `shmavito/models.py`.
- Маршруты: `shmavito/urls.py`.
- Бизнес-логика: `shmavito/views.py`.
- Формы/валидация: `shmavito/forms.py`.
- UI: Django Templates в `templates/`.

## Используемые технологии

- Python 3.13 (по `Dockerfile`)
- Django 5.2.7
- PostgreSQL (`postgres:latest` в `docker-compose.yml`)
- `psycopg2` / `psycopg2-binary`
- `django-admin-interface`, `django-colorfield`
- Docker и Docker Compose

## Предметная область (доменные сущности)

По `shmavito/models.py` в проекте реализованы:
- `Customer`, `CustomerStatus`, `City`
- `Good`, `GoodCategory`, `GoodStatus`
- `GoodImage`, `ImageStatus`
- `Advertisement`, `AdvertisementStatus`
- `Order`, `OrderStatus`
- `CustomerScore`, `Comment`

Это подтверждает, что приложение покрывает базовый цикл аренды: публикация -> поиск -> заказ -> отзыв/оценка.

## Основные пользовательские сценарии

1. Регистрация и вход: `register`, `auth_site`.
2. Публикация товара: `add_good` (+ изображения).
3. Публикация предложения аренды: `add_ad`.
4. Поиск и фильтрация объявлений: `listing` + `AdvertisementSearchForm`.
5. Оформление заказа: `make_order` (с расчетом периода/стоимости).
6. Просмотр своих заказов: `my_orders`.
7. Отмена заказа: `cancel_order` (восстановление доступности дат предложения).
8. Модерация: `moder_good`, `moder_ad`, `approve_*`, `disapprove_*`.
9. Профиль пользователя и отзывы: `user_page`, `order`.

## Процесс развертывания (текущее состояние)

В проекте есть контейнеризация:
- `docker-compose up shmavito_db`
- `docker-compose up shmavito_web`

Команда `shmavito_web` запускает:
1) `migrate`  
2) `loaddata data.json`  
3) `collectstatic --no-input`  
4) `runserver 0.0.0.0:8001`

## Важные ограничения и риски

Несмотря на широкий функционал, текущий репозиторий рассинхронизирован:

- Пакет `core` (включая `core/settings.py`) в дереве проекта не найден, хотя на него ссылается `manage.py`.
- Файл `data.json` не найден, но обязателен в `docker-compose.yml`.
- В `views.py` есть вызовы шаблонов, которых нет в `templates/` (`add_good.html`, `add_ad.html`, `delete_good.html`).
- В `views.py` присутствует дублирование `cancel_order` (первая версия-заглушка и вторая рабочая).
- Функция `comment` содержит невалидные ссылки на переменные (`buyer_id`, `ad_id`) и шаблон `your_template.html`.
- Автотесты фактически отсутствуют (`shmavito/tests.py` пустой).

## Экономические выгоды (для бизнес-контекста)

После стабилизации проекта бизнес получает:
- быстрый запуск нишевого маркетплейса без разработки с нуля;
- снижение операционных затрат за счет автоматизации заказов и модерации;
- рост оборота через повторные сделки и рейтинг доверия;
- возможность масштабирования каталога и географии без линейного роста ручных процессов.

## Приоритетные шаги развития

1. Восстановить/добавить `core/settings.py` и синхронизировать конфигурацию запуска.
2. Привести в порядок обязательные артефакты (`data.json` или изменение boot-пайплайна).
3. Закрыть ошибки шаблонов и удалить битые/дублирующие участки во `views.py`.
4. Добавить минимальные интеграционные тесты на ключевые user flow.
5. Вынести критичную бизнес-логику заказа в сервисный слой.
